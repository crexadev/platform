# Identity Schema

CREXA uses a provider-independent identity layer in Neon PostgreSQL per [ADR 007](decisions/007-authentication-architecture.md).

## Purpose

- CREXA owns a permanent internal UUID for every product identity.
- External authentication identifiers are isolated in a mapping table.
- Clerk user IDs must never appear as foreign keys in product tables.

## Tables

### `users`

CREXA's internal product identity. All future product tables reference `users.id`.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | UUID | Primary key; generated by PostgreSQL |
| `status` | `user_status` enum | `active`, `suspended`, or `deleted` |
| `created_at` | `timestamptz` | Set on insert |
| `updated_at` | `timestamptz` | Application-maintained on update |

### `external_identities`

Maps a CREXA user to an external authentication provider account.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `user_id` | UUID | Foreign key to `users.id`, `ON DELETE CASCADE` |
| `provider` | text | Stable provider identifier (initial value: `clerk`) |
| `provider_user_id` | text | External provider user ID |
| `created_at` | `timestamptz` | Set on insert |
| `updated_at` | `timestamptz` | Application-maintained on update |

## Data ownership

- **Clerk** owns credentials, sessions, and authentication flows.
- **CREXA** owns internal identity records and all product relationships.

## Constraints

| Constraint | Purpose |
| --- | --- |
| `(provider, provider_user_id)` unique | Prevents duplicate mapping of the same external account |
| `(user_id, provider)` unique | One identity per provider per CREXA user for MVP |
| `user_id` index | Resolves all identities for one CREXA user |
| `user_id` → `users.id` cascade delete | Removing a CREXA user removes its identity mappings |

Deleting an external identity row does not delete the CREXA user.

## Lifecycle

- Lazy provisioning will be added in a later approved task.
- Verified Clerk webhooks will be added in a later approved task.
- Deletion and suspension policies require separate approval.
- `updated_at` is application-maintained until a separately approved database-trigger strategy exists.
